# IMKQAS 文档分割策略报告

> 生成日期：2026-06-20 | 基于分支 `master` | 提交 `08ad3b6`

---

## 1. 概述

IMKQAS 医疗知识问答系统的文档分割模块支持四种策略，其中新增的 `medical`（医学文档结构感知）策略是本文档的重点。该策略在传统递归语义分割基础上，引入文档结构树解析、层级元数据注入、表格保护、ICD 编码提取等能力。

**四种策略对比：**

| 策略 | 原理 | 元数据 | 适用场景 |
|------|------|--------|---------|
| `fixed` | 按固定字符数切分 | 仅位置 | 非结构化文本 |
| `recursive` | 按分隔符优先级递归切分 | 仅位置 | 通用文档 |
| `semantic` | 同 recursive（别名） | 仅位置 | 通用文档 |
| **`medical`** | 文档结构树 + 递归降级 | **层级元数据** | 医学指南/知识条目 |

## 2. 架构设计

### 2.1 类关系图

```
RagConfig (配置)
  └─ ChunkConfig.strategy = "medical" ──────────┐
  └─ MedicalChunkConfig (8个可调参数)             │
                                                   │
DocumentProcessorServiceImpl                       │
  └─ chunkText() ─── strategy dispatch ───────────┤
       ├─ "fixed"     → fixedSizeSplit()          │
       ├─ "recursive" → RecursiveTextSplitter     │
       └─ "medical"   → chunkTextMedical() ───────┘
                            │
                            ▼
                     MedicalDocumentSplitter
                            │
                            ├─ Phase 1: 预处理保护
                            │    ├─ protectDosage()    (复用 RecursiveTextSplitter.DOSAGE_PATTERN)
                            │    └─ protectTables()    (表格占位符替换)
                            │
                            ├─ Phase 2: 文档结构解析
                            │    ├─ detectHeaders()    (6类标题正则 + 短行兜底)
                            │    ├─ assignLevel()      (层级深度分配)
                            │    ├─ buildTree()        (父子树构建)
                            │    ├─ computeEndPositions()
                            │    ├─ injectSiblingInfo()
                            │    └─ injectDocumentMeta()
                            │
                            ├─ Phase 3: 沿结构树切分
                            │    ├─ chunkNodeRecursive()
                            │    ├─ chunkLeafContent()
                            │    └─ splitOversized()   → RecursiveTextSplitter 降级
                            │
                            ├─ Phase 4: 同级合并 (mergeSmallSiblings)
                            ├─ Phase 5: 滑动窗口重叠 (addOverlap)
                            └─ Phase 6: 还原 + 过滤 (restoreAndFilter)
```

### 2.2 数据结构

**SectionNode（文档结构树节点）：**
```
level, startChar, endChar, title, parent, children[]
prevSiblingTitle, nextSiblingTitle, documentTopic
```

**SegmentInfo（分片结果——携带完整层级元数据）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `text` | String | 分片文本 |
| `startChar / endChar` | int | 在原文中的字符位置 |
| `sectionPath` | JSON数组 | 章节路径，如 `["治疗方案", "急性发作期治疗"]` |
| `breadcrumb` | String | 面包屑路径，如 `儿童支气管哮喘 > 治疗方案 > 急性发作期治疗` |
| `sectionLevel` | int | 层级深度 (0=正文, 1=一级标题, 2=二级标题, ...) |
| `sectionTitle` | String | 当前章节标题 |
| `parentSection` | String | 父章节标题 |
| `rootSection` | String | 根章节标题 |
| `prevSibling / nextSibling` | String | 前后兄弟章节标题 |
| `documentTopic` | String | 文档主题（从文档标题/前导文本提取） |
| `documentSectionCount` | int | 一级章节总数 |
| `isTable` | boolean | 当前分片是否包含表格 |
| `icdCodes` | List\<String\> | 提取到的 ICD-10 编码 |

## 3. 六阶段流水线详解

### Phase 1：预处理保护

在结构解析前，将容易被误切的模式替换为占位符：
- **剂量信息**：复用 `RecursiveTextSplitter.DOSAGE_PATTERN`，保护如 `沙丁胺醇雾化0.15mg/kg`、`甲泼尼龙1～2mg/kg静注` 等不被切分
- **表格结构**：识别连续管道表格行（`|...|`），当数据行 ≥ `minTableRows`（默认2）时整体替换为占位符，阻止表格被分隔符切断

### Phase 2：文档结构解析

**标题检测**按优先级依次匹配（高优先级先生效）：

| 模式 | 正则示例 | 匹配内容 | 默认层级 |
|------|---------|---------|---------|
| BRACKET_HEADER | `【(【^】]+)】` | `【病因与发病机制】` | 1 |
| MARKDOWN_H1_H2 | `^(#{1,2})\s+(.+)$` | `## 临床表现` | 1 |
| CHINESE_CHAPTER | `^(第[一二三…\d]+[章节篇部分])\s*$` | `第一章` | 1 |
| CHINESE_NUMBERED | `^([一二三四…])、\s*(.+)$` | `一、概述` | 2 |
| DIGIT_HEADER | `^(\d+(?:\.\d+)*)\.?\s+(.+)$` | `9. 儿童支气管哮喘` | 2 |
| COLON_SUBTITLE | `^(.{2,50})[：:]\s*$` | `急性发作期治疗（SABA为核心）：` | 2 |
| SHORT_STANDALONE | 短行 + 后续缩进/列表 | `预防措施` / `紧急处理要点` | 1 |

**树构建**：使用栈维护父节点链，遍历标题列表按层级关系构建父子结构。`computeEndPositions` 自底向上扩展父节点 span 以覆盖所有子节点。

### Phase 3：沿结构树切分

深度优先遍历 SectionNode 树，在节点边界处切分：
- 兄弟节点之间的间隙文本独立成块
- 叶节点内容 ≤ `chunkSize` 时直接成块
- 超长内容回退到 `RecursiveTextSplitter` 递归降级处理
- 根节点前导文本（第一个标题之前的内容）同样保留

### Phase 4：同级合并

相邻的同层级小 chunk（文本长度 < `siblingMergeThreshold`，默认150字符）在不超过 `chunkSize` 的前提下合并，减少语义碎片。

### Phase 5：滑动窗口重叠

与 `RecursiveTextSplitter` 一致：每个 chunk（除首个）的前缀追加前一个 chunk 尾部的 `overlap` 个字符，确保检索时不会因切分边界丢失跨 chunk 上下文。

### Phase 6：还原与过滤

- 还原所有占位符为原始文本
- 丢弃长度 < `minChunkSize`（默认20字符）的碎片
- 在原始文本中重新定位字符偏移
- 检测还原后的表格（`≥ minTableRows` 行管道行标记 `isTable`）
- 提取 ICD-10 编码（正则：`[A-Z]\d{2}(\.\d{1,2})?`）

## 4. 配置参数

### 4.1 通用分块配置（`imkqas.rag.document.chunk`）

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `size` | 500 | 分块大小（字符数） |
| `overlap` | 50 | 滑动窗口重叠字符数 |
| `strategy` | `semantic` | 分块策略，可选 `fixed`/`recursive`/`semantic`/`medical` |
| `min-chunk-size` | 20 | 最小 chunk 长度，低于此值丢弃 |
| `separators` | `\n\n,\n,。,；,，, ,` | 分隔符优先级（降级顺序） |

### 4.2 医学文档专用配置（`imkqas.rag.document.chunk.medical`）

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `section-hierarchy` | true | 是否启用章节层级解析 |
| `table-protection` | true | 是否保护表格不被切分 |
| `icd-detection` | true | 是否提取 ICD-10 编码到元数据 |
| `sibling-context` | true | 是否注入兄弟章节信息（prev/next） |
| `min-section-header-length` | 2 | 最小标题长度（字符），短于此值不识别为标题 |
| `min-table-rows` | 2 | 表格最少数据行数，低于此值不保护 |
| `sibling-merge-threshold` | 150 | 同级小 chunk 合并阈值（字符） |
| `max-depth` | 5 | 最大层级深度，超此深度不再拆分新节点 |

### 4.3 部署方式

配置位于 `application.yml`，生产环境默认使用 `strategy: semantic`。切换到 `medical` 只需修改：

```yaml
imkqas:
  rag:
    document:
      chunk:
        strategy: medical
        medical:
          section-hierarchy: true
          table-protection: true
          icd-detection: true
          sibling-context: true
```

## 5. 元数据检索与生成应用

### 5.1 检索时按层级过滤

chunk 携带的 `section_path` JSON 数组支持按章节路径精确过滤：

```sql
-- 伪SQL：仅检索"治疗方案"章节下的内容
WHERE metadata->>'$.section_path' LIKE '%"治疗方案"%'

-- 仅检索一级标题下的内容（跳过子章节细节）
WHERE metadata->>'$.section_level' = '1'
```

### 5.2 生成时补回上下文

LLM 生成回答时，可以利用面包屑和兄弟关系补充缺失的上下文：

```
给定：
- chunk A: breadcrumb = "儿童支气管哮喘 > 治疗方案 > 急性发作期治疗"
- chunk B: breadcrumb = "儿童支气管哮喘 > 治疗方案 > 长期控制治疗"

生成时自动注入兄弟章节信息：
"当前 chunk 属于【治疗方案】章节，前序章节为【急性发作期治疗】，请确保上下文连贯。"
```

### 5.3 示例 metadata JSON

```json
}
```

## 7. 数据部署指南

### 7.1 从 PDF 到向量库的完整链路

```
PDF上传 → MinerU API 转换(Markdown) → PdfPreCheck(扫描件判断)
    → PageLevelClassifier(页面级分类) → TextCleaner(清洗)
    → DocumentProcessorServiceImpl.chunkText()
        strategy="medical" → MedicalDocumentSplitter.splitWithMetadata()
            → 7个SegmentInfo(带层级元数据)
    → EmbeddingService.embed() → text-embedding-v3(1024维)
    → Milvus INSERT(含metadata JSON) → medical_documents collection
```

### 7.2 向量库字段设计

Milvus `medical_documents` collection 的核心字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | Int64 | 主键 |
| `document_id` | Int64 | 所属文档ID |
| `chunk_index` | Int32 | chunk 序号 |
| `content` | VarChar | 分片文本 |
| `embedding` | FloatVector(1024) | 嵌入向量 |
| `metadata` | JSON | 层级元数据（section_path, breadcrumb, icd_codes 等） |

### 7.3 检索时利用层级元数据

```
1. 用户查询 → query embedding (1024维)
2. Milvus ANN 搜索 → top-30 candidates
3. 关键词 Lucene 搜索 → top-30 candidates
4. RRF 融合 → 候选集
5. Reranker 精排 → top-5
6. 可选：按 metadata.section_path 过滤/加权
7. LLM 生成（注入 breadcrumb 上下文）
```

## 8. 后续优化方向

1. **前端集成**：在 `KnowledgeView.vue` 的 chunk 浏览器中展示层级面包屑和 ICD 编码标记
2. **检索加权**：在 RRF 融合阶段对同文档高 section_level 的 chunk 降权（优先返回大纲级内容）
3. **自动策略选择**：在 `PdfPreChecker` 阶段根据文档类型自动选择 `medical` vs `semantic` 策略
4. **更多标题模式**：支持罗马数字（I./II.）、数字加括号（1)/2)）等额外标题格式
5. **跨文档关联**：利用 ICD 编码实现知识条目之间的交叉引用
